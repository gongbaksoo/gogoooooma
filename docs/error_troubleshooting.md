## 1. 일평균 계산 관련 에러 (가장 핵심적인 로직 오류)

### 🚨 문제 상황 1: 최근 3개월 계산 시 당월이 포함되는 문제
- **증상**: "최근 3개월" 데이터에 데이터가 불완전한 당월(진행 중인 달)이 포함되어 월평균 및 일평균 매출이 비정상적으로 낮게 계산됨.
- **원인**: `get_monthly_summary` 함수에서 `last_3_months = unique_months[-3:]` 로 코딩되어 있어 리스트의 마지막 요소(당월)가 포함됨.
- **해결 방안**: 당월을 제외한 이전 3개월을 가져오도록 슬라이싱 로직 수정.
  ```python
  # 수정 전
  last_3_months = unique_months[-3:]
  
  # 수정 후 (당월 제외, 과거 3개월)
  last_3_months = unique_months[-4:-1]
  ```

### 🚨 문제 상황 2: 전월 및 과거 달의 일평균이 실제 달력 일수가 아닌 데이터 존재 일수로 나누어지는 문제
- **증상**: 1월(31일) 데이터 중 주말 등 매출이 없는 날짜가 제외된 채 23일치의 데이터만 CSV에 기록되어 있을 때, 1월 총 매출을 31이 아닌 23으로 나누어 일평균 매출이 과다하게 높게 계산됨. (당월과 과거 달의 계산 기준 불일치)
- **원인**: 모든 달에 대해 `days_count = monthly_data[date_col].nunique()`를 적용하여 실제 데이터가 존재하는 일수만으로 나누었음.
- **해결 방안**:
  1. `get_calendar_days(month_str)` 헬퍼 함수를 추가하여 파이썬 내장 `calendar.monthrange`를 통해 과거 달의 실제 달력 일수를 구함.
  2. 당월은 `nunique()`(실제 데이터 일수)를 사용하고, 과거 달(전월, 최근 3개월, 전년 동월)은 달력 일수를 사용하여 나누도록 조건 분기 추가.

### 🚨 문제 상황 3: 최근 3개월 '누적' 매출이 표기되는 문제
- **증상**: 테이블의 "최근 3개월" 열에 누적 매출(예: 14억)이 표기되어 전월 누적(예: 4억) 등과 스케일이 맞지 않아 비교가 어려움.
- **원인**: 최근 3개월의 단순 합산을 반환했음. 유저가 원한 것은 최근 3개월의 "월평균" 매출이었음.
- **해결 방안**: 누적 금액을 3으로 나누어 월평균을 계산하도록 프론트/백엔드 수정.
  ```python
  # 수정 후 로직
  "last_3months_total": int(last_3_stats['total'] / 3)
  ```

## 2. 데이터 로드 및 파싱 (캐싱 관련 에러)

### 🚨 문제 상황: 과거 데이터만 계속 로드되는 현상 (캐시 충돌)
- **증상**: 2026년 2월의 최신 CSV 파일을 업로드했는데도 불구하고, 대시보드에서는 "2021년 1월 ~ 2022년 7월" 등의 과거 데이터만 계속해서 렌더링됨.
- **원인**: 백엔드 성능 최적화를 위해 도입된 **Parquet 기반 파일 캐싱**(`uploads/cache/*.parquet`) 및 인메모리 캐시(`df_cache`) 메커니즘에서, 같은 이름의 CSV를 덮어썼을 때 캐시가 갱신되지 않거나, 다른 파싱 규칙이 적용된 과거 캐시가 반환되는 이슈 발생.
- **해결 방안**:
  1. 관리자용 캐시 클리어 API 엔드포인트 `/api/cache/clear`를 통해 서버 내 Parquet 캐시 파일과 메모리 딕셔너리를 강제로 삭제.
  2. 향후 파일명에 timestamp를 붙이거나 파일 해시값을 검증하여 캐시 무효화(Cache Invalidation) 로직 강화 필요.

## 3. UI 및 렌더링 관련 에러

### 🚨 문제 상황: "전체 매출" 테이블 행 파란선 스타일 이슈
- **증상**: 전체 매출 행 왼쪽에 파란색 세로선(Accent Indicator)이 존재하여 유저가 제거를 요청함.
- **원인**: `SalesSummary.tsx`에서 `isTotal` 조건부 렌더링으로 `<div className="w-1 h-6 bg-blue-600 ...">` 컴포넌트가 하드코딩 되어 있었음.
- **해결 방안**: 해당 `div` 태그를 제거하고 배포 진행.

### 🚨 문제 상황: 알림 카드(Alert Card) 일평균 계산 불일치
- **증상**: 메인 대시보드 테이블의 일평균 계산 로직은 고쳤으나, "알림 카드(Alert Card)" 섹션에 나타나는 일평균 성장률(%) 수치가 여전히 과거 달을 `nunique()`(실제 일수)로 나누어 이상하게 표시됨.
- **원인**: 테이블용 함수(`get_monthly_summary`)와 알림 카드용 분석 함수(`analyze_sales_performance`)가 분리되어 있어, 로직 변경이 한쪽에만 반영되었음.
- **해결 방안**: `analyze_sales_performance` 내부의 `get_period_stats` 함수에서도 동일하게 당월인지 판별(`pm == curr_month`)하여 당월은 `nunique()`, 과거 월은 `calendar` 달력 일수를 적용하도록 수정.

### 🚨 참고: Railway 컨테이너의 디스크는 휘발성
- **현황**: 백엔드는 Railway에서 실행되며 컨테이너 디스크는 재배포/재시작 시 초기화됨. 권위 있는(authoritative) 저장소는 PostgreSQL의 `uploaded_files` 테이블(BLOB)이며, `uploads/`와 `uploads/cache/*.parquet`은 모두 캐시 성격이라 매 배포마다 사라질 수 있음.
- **영향**: 데이터 무결성에는 문제 없음(`ensure_file_on_disk`가 DB → 디스크 동기화). 다만 parquet 캐시가 매 배포마다 0에서 시작하므로 첫 요청에서 원본 파일을 다시 파싱함 → 일시적 응답 지연.
- **권장**: 반복 재배포 시 첫 요청 지연이 문제면 Railway 영구 볼륨 활성화 또는 워밍업 엔드포인트 추가 검토.

## 4. 향후 유지보수를 위한 권장 사항

1. **테스트 데이터 분리**: 특정 케이스(주말이 빠진 데이터, 한 달의 중간까지만 있는 데이터) 등을 가진 `mock_sales_data.csv`를 만들어 일평균 계산 로직이 맞는지 정기적으로 검증하는 테스트 코드 작성이 필요합니다.
2. **상태 관리/세션 유지**: 현재 유저가 페이지를 새로고침하면 어떤 파일을 조회하고 있었는지 초기화됩니다. 파일 선택 상태를 브라우저 `localStorage`나 URL Query Parameter에 저장하도록 프론트엔드 개선이 필요합니다.
3. **캐시 관리**: 파일명 기반 캐싱은 동일 파일명 덮어쓰기 시 버그를 유발합니다. 파일 업로드 시 서버에서 부여한 고유 UUID나 파일의 SHA256 해시를 캐시 키로 사용하도록 백엔드 구조를 개편해야 합니다.
